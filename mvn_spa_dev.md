# Maven AEM Project — Vue SPA Marketing Site

End-to-end build of `spa-mvn/`: a production-shaped AEM project generated by the Adobe AEM Project Archetype, with the `ui.frontend` module rewritten in Vue 3 + Vite, rendering a multi-section Adobe-style marketing home page. Deployed to the author container via `mvn -PautoInstallPackage`.

This is **Option B** from `spa_example.md` — what an AEM team would actually ship — captured as a working walkthrough including every gotcha.

## Result

- `http://localhost:4502/content/spa-mvn.html` → redirects to `/content/spa-mvn/us/en.html`
- 200 OK, `text/html`, Vue SPA mounts on `#spa-mvn-app`
- Clientlibs served at `/etc.clientlibs/spa-mvn/clientlibs/clientlib-site.lc-<hash>-lc.min.{js,css}`
- Page sections: AppHeader, HeroSection, LogoStrip, FeatureGrid (6 cards), StatsBar, CTABanner (with form), AppFooter

## Toolchain

| Tool | Version | Purpose |
|---|---|---|

## Step-by-step what we did

### 1. Install JDK + Maven

```bash
sudo apt-get update
sudo apt-get install -y openjdk-11-jdk-headless maven
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
java --version          # OpenJDK 11.0.30
mvn --version           # Apache Maven 3.8.7
```

Adobe's Cloud SDK runs on JDK 11; the archetype expects 11. JDK 17 may work for some builds but Adobe's reference setup is 11.

### 2. Generate the project from the archetype

```bash
cd /home/mli/aem-docker

mvn -B org.apache.maven.plugins:maven-archetype-plugin:3.2.1:generate \
  -D archetypeGroupId=com.adobe.aem \
  -D archetypeArtifactId=aem-project-archetype \
  -D archetypeVersion=49 \
  -D appTitle="SPA MVN" \
  -D appId="spa-mvn" \
  -D groupId="com.example.spamvn" \
  -D artifactId="spa-mvn" \
  -D version="0.0.1-SNAPSHOT" \
  -D package="com.example.spamvn" \
  -D frontendModule=general \
  -D includeExamples=n \
  -D includeErrorHandler=n \
  -D includeDispatcherConfig=n \
  -D aemVersion=cloud \
  -D singleCountry=y \
  -D country=us \
  -D language=en
```

**⚠️ Why pin `maven-archetype-plugin:3.2.1`** — the default 3.4.x runs Velocity on EVERY file in the archetype, including `.gif` binaries. You get:

```
[ERROR] Encountered "<EOF>" at .../busy-state.gif[line 283, column 89]
```

Older plugin 3.2.1 skips binary files. Use that one.

### 4. Replace webpack scaffold with Vite + Vue 3

The archetype's `ui.frontend/` ships with webpack + TypeScript + a non-trivial config. We swapped it for Vite + Vue while keeping the same downstream pipeline (`aem-clientlib-generator` copies Vite output into `ui.apps/.../clientlib-site/`).

**Files removed:**

```bash
cd ui.frontend
rm -f webpack.common.js webpack.dev.js webpack.prod.js \
      tsconfig.json .babelrc .eslintrc.js .eslintignore \
      package-lock.json
```

**New `ui.frontend/package.json`:**

```json
{
  "name": "spa-mvn-frontend",
  "version": "1.0.0",
  "private": true,
  "type": "module",
  "scripts": {
    "build": "vite build",
    "prod": "vite build && clientlib clientlib.config.cjs --verbose",
    "dev":  "vite build --mode development && clientlib clientlib.config.cjs --verbose"
  },
  "dependencies":    { "vue": "^3.5.0" },
  "devDependencies": {
    "@vitejs/plugin-vue":     "^5.2.0",
    "aem-clientlib-generator": "^1.8.0",
    "vite":                    "^5.4.0"
  }
}
```

**New `ui.frontend/vite.config.js`:**

```js
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import { resolve } from 'path'

export default defineConfig({
  plugins: [vue()],
  build: {
    outDir: resolve(__dirname, 'dist'),
    emptyOutDir: true,
    cssCodeSplit: false,
    rollupOptions: {
      input: resolve(__dirname, 'src/main.js'),
      output: {
        dir: resolve(__dirname, 'dist/clientlib-site'),
        entryFileNames: 'js/site.js',
        chunkFileNames: 'js/[name].js',
        assetFileNames: (info) =>
          info.name?.endsWith('.css') ? 'css/site.css' : 'resources/[name][extname]'
      }
    }
  }
})
```

The output layout (`dist/clientlib-site/js/site.js`, `.../css/site.css`) matches what `aem-clientlib-generator` expects to copy.

**Renamed `clientlib.config.js` → `clientlib.config.cjs`** because we set `"type": "module"` in `package.json`. Node refuses to `require()` a `.js` file under an ESM package; `.cjs` is the escape hatch. The `clientlib` CLI accepts the config path as a positional arg.

Simplified `clientlib.config.cjs` (single clientlib instead of the archetype's two):

```js
const path = require('path');
const BUILD_DIR = path.join(__dirname, 'dist');
const CLIENTLIB_DIR = path.join(__dirname, '..', 'ui.apps', 'src', 'main', 'content',
                                'jcr_root', 'apps', 'spa-mvn', 'clientlibs');

module.exports = {
  context: BUILD_DIR,
  clientLibRoot: CLIENTLIB_DIR,
  libs: [{
    allowProxy: true,
    serializationFormat: 'xml',
    cssProcessor: ['default:none', 'min:none'],
    jsProcessor:  ['default:none', 'min:none'],
    name: 'clientlib-site',
    categories: ['spa-mvn.site'],
    assets: {
      js:  { cwd: 'clientlib-site/js',  files: ['**/*.js'],  flatten: false },
      css: { cwd: 'clientlib-site/css', files: ['**/*.css'], flatten: false }
    }
  }]
};
```

### 5. Bump Node version in parent pom

The archetype pins `frontend-maven-plugin` to download **Node 16** — too old for Vite 5 (which uses `crypto.getRandomValues` API added in Node 18). The error is:

```
TypeError: crypto$2.getRandomValues is not a function
```

Fix — in **`spa-mvn/pom.xml`** (the parent), find the `frontend-maven-plugin` `<configuration>` in `<pluginManagement>` and change:

```xml
<nodeVersion>v16.17.0</nodeVersion>
<npmVersion>8.15.0</npmVersion>
```

to:

```xml
<nodeVersion>v20.18.1</nodeVersion>
<npmVersion>10.8.2</npmVersion>
```

The plugin downloads this Node into `ui.frontend/node/` on first run; that node is used for all subsequent `npm` invocations.

### 6. Override `page.html` in the page component

Default page rendering (`core/wcm/components/page/v3/page`) walks the responsive grid and renders the authored components. For a SPA where Vue draws everything, we replace the body with a single mount div + clientlib include.

**`ui.apps/src/main/content/jcr_root/apps/spa-mvn/components/page/page.html`:**

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8"/>
    <meta name="viewport" content="width=device-width, initial-scale=1.0"/>
    <title>${currentPage.title || 'SPA MVN'}</title>
    <sly data-sly-use.clientlib="/libs/granite/sightly/templates/clientlib.html"
         data-sly-call="${clientlib.css @ categories='spa-mvn.site'}"></sly>
</head>
<body>
    <div id="spa-mvn-app"></div>
    <sly data-sly-use.clientlib="/libs/granite/sightly/templates/clientlib.html"
         data-sly-call="${clientlib.js @ categories='spa-mvn.site'}"></sly>
</body>
</html>
```

Because we override the top-level `page.html`, the existing `customheaderlibs.html` / `customfooterlibs.html` files (which reference a non-existent `spa-mvn.base` category) are no longer called. The page renders only what's in our HTL.

Note: this disables AEM Sites authoring on the page — you can no longer drag components from the side panel because Vue owns the DOM. That's the trade-off when you go SPA. The Adobe SPA Editor SDK (React/Angular only) reintroduces authoring via a JSON-driven component model; for Vue you either accept "no authoring" or hand-build a Vue↔AEM editor bridge.

### 7. Vue source — the marketing page

```
ui.frontend/src/
├── main.js                 createApp(App).mount('#spa-mvn-app')
├── App.vue                 composes all sections in order
├── styles/global.css       resets, container, .section helpers, color tokens
└── components/
    ├── AppHeader.vue       Sticky nav with logo, links, "Start free trial" CTA, mobile toggle
    ├── HeroSection.vue     Dark gradient bg, eyebrow + h1 + lede + dual CTAs + meta row
    ├── LogoStrip.vue       Customer logo row ("Trusted by")
    ├── FeatureGrid.vue     3×2 grid wrapping FeatureCard
    ├── FeatureCard.vue     Icon + title + body + "Learn more →" — hover lifts the card
    ├── StatsBar.vue        4 metric tiles on dark band
    ├── CTABanner.vue       Gradient card with email capture form (reactive disabled state)
    └── AppFooter.vue       4-column link grid + brand block + copyright
```

Color palette (avoid trademarks — generic blue + red gradient):
- Primary blue:  `#1473e6`
- Accent red:    `#ff3b30`
- Dark surface:  `#0c0e2c`
- Body:          `#1a1a1a` on white / `#fafafa`

Component patterns used:
- `<script setup>` syntax — no `defineComponent`, no boilerplate
- Scoped CSS in every `.vue` file — no global selectors clash
- `:deep(svg)` in FeatureCard for styling inline SVG children
- `computed()` in CTABanner for the dynamic button label (writing complex strings inline in `{{ }}` triggers the parser bug below)

### 8. Build and deploy

```bash
export JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
cd /home/mli/aem-docker/spa-mvn
mvn -PautoInstallPackage clean install
```

Phases executed (in order):

| Module | What happens |
|---|---|
| `core` | Compile Java → OSGi bundle `.jar` |
| `ui.frontend` | npm ci → `vite build` → `aem-clientlib-generator` copies `dist/clientlib-site/{js,css}/*` into `ui.apps/.../clientlibs/clientlib-site/` |
| `ui.apps.structure` | Build empty structural content package |
| `ui.apps` | filevault-package-maven-plugin packages `/apps/spa-mvn/*` → `.zip` |
| `ui.content` | filevault-package-maven-plugin packages `/content/spa-mvn/*` → `.zip` |
| `ui.config` | OSGi config bundle |
| `all` | aggregator that bundles `ui.apps + ui.content + ui.config + core.jar` into one installable `all-0.0.1-SNAPSHOT.zip` |
| **`autoInstallPackage` profile** | POSTs the `all` package to `http://localhost:4502/crx/packmgr/service.jsp` |
| `it.tests` | runs Sling integration tests against the live author |
| `ui.tests` | Cypress UI tests |

## Redeploy workflow

Iteration loop after the first build:

| What changed | Command | Time |
|---|---|---|
| Vue source only | `mvn -PautoInstallPackage -pl ui.frontend,ui.apps,all -am install` | ~25 s |
| HTL template only | `mvn -PautoInstallPackage -pl ui.apps,all -am install` | ~10 s |
| Java code in core | `mvn -PautoInstallPackage -pl core,all -am install` | ~15 s |
| Anything (safe default) | `mvn -PautoInstallPackage clean install` | ~60 s |

For frontend-only inner loop without the full Maven roundtrip:

```bash
cd ui.frontend
npm run prod                                                # Vite → clientlib copy
curl -u admin:admin -T dist/clientlib-site/js/site.js \
  http://localhost:4502/apps/spa-mvn/clientlibs/clientlib-site/js/site.js
curl -u admin:admin -T dist/clientlib-site/css/site.css \
  http://localhost:4502/apps/spa-mvn/clientlibs/clientlib-site/css/site.css
```

But the maven flow is preferred — it keeps the artifact reproducible from source.

## Promotion to other environments

The shippable artifact is **`all/target/spa-mvn.all-0.0.1-SNAPSHOT.zip`** (~few MB).

```
Dev (this box)
    │  mvn -PautoInstallPackage  →  /crx/packmgr/service.jsp
    │
Stage / Prod
    │  Upload all-*.zip via:
    │   - Cloud Manager pipeline (production)
    │   - Package Manager UI at /crx/packmgr/index.jsp
    │   - curl multipart POST to /crx/packmgr/service.jsp
```

A single artifact promotes through environments — no per-environment rebuild. Runmode-specific behavior (auth credentials, feature flags) lives in `ui.config/` as filtered OSGi configs.

## Pitfalls hit & resolved

| Symptom | Cause | Fix |
|---|---|---|
| `Encountered "<EOF>" at .../busy-state.gif[line 283, column 89]` during archetype generation | Default `maven-archetype-plugin` 3.4.x tries Velocity-parse on binary files | Pin plugin to `3.2.1`: `mvn org.apache.maven.plugins:maven-archetype-plugin:3.2.1:generate ...` |
| `npm ci: ENOENT package-lock.json` during mvn build | Lock file deleted; `npm ci` requires it | Run `npm install` once in `ui.frontend/` to regenerate `package-lock.json`, commit it |
| `ReferenceError: require is not defined in ES module scope` when running `clientlib` | `"type": "module"` in package.json + `.js` config file using `require()` | Rename `clientlib.config.js` → `clientlib.config.cjs`, pass as positional arg |
| `TypeError: crypto$2.getRandomValues is not a function` during `vite build` | Node 16 (downloaded by frontend-maven-plugin) too old for Vite 5 | Bump `<nodeVersion>v20.18.1</nodeVersion>` in parent `pom.xml` |
| `Error parsing JavaScript expression: Unexpected token, expected ":"` at `{{ submitted ? '...' : '...' }}` | HTML entity `&apos;` inside a JS string literal inside a `{{ }}` interpolation; Vue tries to parse the entity literally | Move the conditional to a `computed()` ref; bind `{{ buttonLabel }}` |
| Page authoring no longer works | Override of `page.html` skips the Core Components page renderer that's authoring-aware | Expected — SPA pages can't be authored without the SPA Editor SDK. Either accept it (this project) or build a bridge (out of scope for Hello-World-grade work) |

## Comparison with the quick path

See `spa_example.md` for the curl-only Option A (`spa-hello/` in this repo). Side-by-side:

| | `spa-hello/` (Option A) | `spa-mvn/` (Option B) |
|---|---|---|
| Build tools needed | `node`, `curl` | JDK 11, Maven, Node (via plugin), npm |
| Project files | 9 hand-written | ~80 generated + 12 modified/added |
| Deploy mechanism | `curl -T` per file | `mvn -PautoInstallPackage` |
| Deploy artifact | none — direct JCR writes | versioned `all-*.zip` package |
| First build time | ~10 s (npm install + 1 vite build) | ~3 min (npm + AEM Uber Jar download + 10 modules) |
| Incremental build | ~1 s vite + 2× curl PUT | ~25 s mvn `-pl ui.frontend,ui.apps,all -am` |
| Promotable to stage/prod | no — would re-curl | yes — same `.zip` to all envs |
| Cloud Manager / CI integration | no | yes |
| Java backend possible | no | yes (`core/` module) |
| Right for | spike, JCR experiment, demo | real project |
